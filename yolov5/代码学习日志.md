## 2. 目录结构理解（当前版本）

### 2.1 `models/`
- `models/yolo.py`：模型组网核心，包含 `Detect`（检测头）和 `Segment`（分割头）实现。
- `models/common.py`：基础模块库（如 `Conv`、`C3`、`SPPF`、`Concat` 等）。
- `models/experimental.py`：实验性模块与部分辅助逻辑。
- `models/tf.py`：TensorFlow/Keras/TFLite 相关适配与导出实现。
- `models/yolov5*.yaml`：标准检测模型结构配置。
- `models/hub/`：更多模型配置 YAML（Hub/结构变体），不是“基础模块代码”。
- `models/segment/`：分割模型 YAML 配置，不是检测头实现代码。

### 2.2 `utils/`
- 核心优先阅读：
  - `utils/dataloaders.py`：数据加载
  - `utils/augmentations.py`：数据增强

  - `utils/torch_utils.py`：PyTorch 工具（设备、EMA 等）
- 其他：
  - `utils/loggers/`：日志后端
  - `utils/aws`、`utils/docker`、`utils/flask_rest_api`、`utils/google_app_engine`：部署/工程化支持
  - `utils/segment/`：分割专用工具

### 2.3 根目录关键文件

## 4. DDP 相关变量理解（已确认）
- `LOCAL_RANK = int(os.getenv("LOCAL_RANK", -1))`
  - 本机进程编号（通常对应本机 GPU 序号）
- `RANK = int(os.getenv("RANK", -1))`
  - 全局进程编号
- `WORLD_SIZE = int(os.getenv("WORLD_SIZE", 1))`
  - 全局总进程数
- `GIT_INFO = check_git_info()`
  - 记录当前仓库版本信息（分支/commit/状态）用于复现与日志

结论：这些变量主要用于分布式训练；单卡普通训练时通常走默认值（如 `RANK=-1`、`WORLD_SIZE=1`）。

## 6. 回调机制（Callbacks）学习记录

### 6.1 `Callbacks` 是什么
- `Callbacks` 是事件钩子管理器，用来在训练/验证流程的固定阶段触发自定义函数。
- 常见 hook：`on_train_start`、`on_train_batch_end`、`on_fit_epoch_end`、`on_model_save` 等。
- 作用：把日志、可视化、平台上报等功能和主训练流程解耦。

### 6.2 “钩子（hook）”怎么理解
- 可理解为主流程预留的“插口/触发点”。
- 程序运行到某阶段就执行挂在该阶段的函数，概念上接近“事件触发器”。
- 与硬件中断不同：钩子是代码里显式设计的扩展点，可注册多个回调，受程序控制。

### 6.3 `register_action()` 理解
- 功能：往某个 hook 上注册一个回调函数。
- 关键点：
  - 校验 hook 是否存在：`hook in self._callbacks`
  - 校验回调是否可调用：`callable(callback)`
  - 注册：`append({"name": name, "callback": callback})`
- 结论：`register_action()` 负责“登记”，不负责执行。

### 6.4 `callable(...)` 理解
- 用于判断对象是否可被调用（能否写成 `obj(...)`）。
- 常见可调用对象：函数、类、实现 `__call__` 的对象。

### 6.5 `run()` 理解
- 功能：触发指定 hook，并执行该 hook 下注册的所有回调。
- 参数：
  - `hook`：事件名
  - `*args`：位置参数（元组）
  - `thread=False`：是否用守护线程异步执行回调
  - `**kwargs`：关键字参数（字典）
- 执行模式：
  - `thread=False`：同步执行
  - `thread=True`：每个回调起一个线程异步执行

### 6.6 `*args` 和 `**kwargs` 的实际含义
- 没有固定内容，取决于 `callbacks.run(...)` 调用处传了什么。
- `*args`：按顺序传参，常见如 `epoch`、`batch_idx`、`model`、`imgs`、`targets`。
- `**kwargs`：按键名传参，常见如 `epoch=...`、`last=...`、`best=...`、`final_epoch=...`。
- 区别：`args` 按位置取，`kwargs` 按名字取。

## 7. `train.py` 训练流程（细版）

### 7.1 入口阶段
1. `parse_opt()`：解析命令行参数，得到 `opt`。
2. `main(opt)`：参数检查、`resume/evolve` 分支、保存目录处理。
3. `select_device(...)`：选择 CPU/GPU/MPS，处理 DDP 环境变量与合法性检查。

### 7.2 训练前准备
1. 进入 `train(hyp, opt, device, callbacks)`，触发 `on_pretrain_routine_start`。
2. 创建权重目录，保存 `hyp.yaml`、`opt.yaml`（非 evolve）。
3. 主进程初始化 `Loggers`，并把 logger 的 hook 方法注册到 `callbacks`。
4. `check_dataset(data)`：读取并校验数据集配置，解析绝对路径，必要时自动下载。

### 7.3 模型与优化器初始化
1. 构建模型 `Model(...)`，按需加载预训练权重（交集加载）。
2. 按 `--freeze` 冻结指定层（`requires_grad=False`）。
3. 计算 `gs/imgsz`，确保输入尺寸满足步长约束。
4. 创建 `optimizer`、`scheduler`、`EMA`、`GradScaler`、`EarlyStopping`、`ComputeLoss`。
5. `resume` 时恢复优化器、EMA、起始 epoch、最佳指标等状态。

### 7.4 数据加载器阶段
1. 创建 `train_loader`（DDP 下使用每 rank 子 batch）。
2. 校验标签类别范围（最大类别 id 必须 `< nc`）。
3. 主进程创建 `val_loader`，可选执行 `check_anchors`。
4. 触发 `on_pretrain_routine_end`。
5. DDP 模式下 `model = smart_DDP(model)`。

### 7.5 正式训练循环
1. 训练开始：绑定 `model.nc/model.hyp/model.names/class_weights`，触发 `on_train_start`。
2. Epoch 循环：
   - `on_train_epoch_start`，`model.train()`
   - 可选 image-weights 重采样
   - DDP 下 `sampler.set_epoch(epoch)`
3. Batch 循环：
   - `on_train_batch_start`
   - 数据搬运与归一化
   - warmup 动态调整 LR/动量/累积步数
   - 可选 multi-scale
   - 前向、损失计算、AMP 反向
   - 到累积步后执行 `optimizer.step()`、梯度裁剪、EMA 更新
   - 主进程更新进度条并触发 `on_train_batch_end`

### 7.6 每轮验证与保存
1. `scheduler.step()` 更新学习率。
2. 主进程触发 `on_train_epoch_end`，并用 EMA 模型执行 `validate.run(...)`。
3. 计算 `fitness`，更新 `best_fitness`，执行早停判断。
4. 触发 `on_fit_epoch_end`。
5. 保存 `last.pt`、`best.pt`（及可选 `epochX.pt`），触发 `on_model_save`。
6. DDP 下广播 `stop` 标记，保证各 rank 同步停止。

### 7.7 收尾阶段
1. 主进程打印训练总耗时。
2. 对 `last/best` 执行 `strip_optimizer` 减小体积。
3. 对 `best.pt` 做最终验证（COCO 时会有额外逻辑）。
4. 触发 `on_train_end`。
5. `torch.cuda.empty_cache()`，返回最终 `results`。
